Moduł 2.1

Czym jest MCP

Model Context Protocol - protokół komunikacji agenta z zewnętrznymi narzędziami

Zasada działania

  • MCP server wystawia narzędzia (tools) dla agenta
  • Agent wywołuje narzędzia w trakcie pracy, gdy ich potrzebuje
  • Serwer działa jako osobny proces, stale dostępny
  • Konfiguracja w pliku .mcp.json lub settings

Do czego służy

  • Dostęp do dokumentacji bibliotek (Context7)
  • Sterowanie przeglądarką (Playwright MCP)
  • Integracja z bazami danych, API zewnętrznymi
  • Każde narzędzie, które agent może wywołać programowo
Idea: zamiast wklejać dokumentację w prompt, agent sam pobiera aktualną dokumentację przez MCP, gdy jej potrzebuje.

Moduł 2.1

Dlaczego coraz częściej Skills zamiast MCP

MCP - kiedy jeszcze sensowne

  • Stała integracja z usługą (baza danych, API)
  • Narzędzie, którego używasz codziennie, przez cały projekt
  • Playwright MCP - sterowanie przeglądarką w E2E
  • Context7 - dokumentacja na żądanie

Skills - kiedy lepsze

  • Powtarzalny workflow (ciąg kroków, nie jedno narzędzie)
  • Bundle instrukcji + skryptów w jednym miejscu
  • Łatwiejsze do wersjonowania i udostępniania
  • Mniej narzut niż utrzymywanie serwera MCP
Trend: agenty przechodzą od "stałych serwerów MCP" do "Skills + CLI wywoływane na żądanie". MCP zostaje dla trwałych integracji, Skills przejmują workflows.

Konfiguracja MCP: .mcp.json i settings

Plik .mcp.json

Konfiguracja serwerów MCP w projekcie. Definiuje, które serwery są dostępne i jak je uruchomić.

{ "mcpServers": { "context7": { "command": "cmd", "args": ["/c", "npx", "-y", "@upstash/context7-mcp@latest"] }, "playwright": { "command": "cmd", "args": ["/c", "npx", "-y", "@playwright/mcp@latest"] } } }

Poziomy konfiguracji

  • .mcp.json w katalogu projektu - per projekt
  • Settings globalne - dostępne w każdym projekcie
  • Agent wczytuje konfigurację przy starcie
Sprawdź: użyj komendy /mcp w CLI, aby zobaczyć aktywne serwery MCP i ich status - tak zweryfikujesz, że konfiguracja zadziałała.

Moduł 2.1 | Narzędzia

Context7 - aktualna dokumentacja

Co to jest

  • Serwer MCP dostarczający aktualną dokumentację bibliotek
  • Agent pobiera dokumentację na żądanie, gdy pisze kod
  • Ogranicza ryzyko użycia nieaktualnego lub wymyślonego API; wynik nadal trzeba zweryfikować
  • Działa dla dowolnej biblioteki z indeksem Context7

Dlaczego ważne

Bez Context7 agent opiera się na wiedzy z treningu, która może być nieaktualna. Z Context7 agent ma dostęp do realnej, bieżącej dokumentacji.

Idealne dla naszego projektu: dokumentacja Vercel AI SDK, React, TypeScript - wszystko aktualne.

Praktyka: gdy agent pisze kod z biblioteką, poproś go o sprawdzenie dokumentacji przez Context7.

Moduł 2.1 | Narzędzia

Playwright MCP - przeglądarka w rękach agenta

Sterowanie przeglądarką

Agent otwiera strony, klika, czyta DOM, robi zrzuty ekranu. Pełna automatyzacja przeglądarki przez MCP.

Design Guidelines

Reverse-engineering designu dowolnej strony: kolory, typografia, układy. Wyciąganie design tokens do własnej aplikacji.

Testy E2E

Automatyczne testy end-to-end. Agent wchodzi na aplikację, klika, weryfikuje czy funkcja działa. Bez mockowania.

Manualne testy agenta

Agent testuje aplikację jak człowiek: otwiera, przechodzi przez flow i sprawdza rezultaty. W E2E używa realnego stosu, w tym prawdziwego LLM.

W naszym procesie: Playwright MCP jest kluczowe na dwóch etapach - Design Guidelines (RAZEM 2.1) oraz testy manualne aplikacji (moduł 2.4 i dzień 3).

Moduł 2.1 | Ważny trend

CLI rośnie vs MCP

Klasycznie: serwer MCP stale włączony

  • Serwer MCP działa cały czas, nawet gdy niepotrzebny
  • Narzut: proces, pamięć, konfiguracja utrzymania
  • Dobre dla integracji, których używasz stale

Trend: CLI wywoływane przez agenta

  • Agent wywołuje narzędzie CLI na żądanie, gdy potrzebuje
  • Brak stale działającego serwera - mniej narzutu
  • Narzędzie startuje, robi robotę, kończy
  • Łatwiejsze debugowanie i izolacja
Praktyczna konsekwencja: coraz więcej narzędzi dostępnych jako CLI wywoływane przez agenta, zamiast trwale skonfigurowanego serwera MCP. Wybieraj świadomie - MCP dla stałych integracji, CLI/Skills dla jednorazowych i workflow.

Moduł 2.1

RAZEM Design Guidelines przez Playwright MCP

Robimy to razem - ja pokazuję, Ty powtarzasz u siebie (możesz na własnym pomyśle aplikacji)

Zrób u siebie

  • Wybierz stronę (firmową uczestnika lub dowolną)
  • Użyj Playwright MCP do zbadania jej designu
  • Wyciągnij design tokens: kolory, typografia, spacing
  • Zrób zrzuty ekranu kluczowych komponentów
  • Zapisz notatkę: co warto przenieść do własnej aplikacji

Zanim ruszymy dalej, upewnij się, że

  • Masz listę tokenów kolorów (paleta)
  • Masz tokeny typografii (fonty, rozmiary, wagi)
  • Masz min. 3 zrzuty ekranu kluczowych komponentów
  • Wynik jest zapisany w repo (design/), nie tylko w pamięci agenta
Prompty podsyłam na czacie - ale zachęcam do eksperymentowania z własną stroną.
Wskazówka: poproś agenta: "Zbadaj design strony [URL] przez Playwright MCP. Wyciągnij design tokens i zrób zrzuty ekranu kluczowych elementów."

Co mówić

  • Hands-on - każdy bada własną stronę przez Playwright
  • Wynik (design tokens) wykorzystamy w module 2.4 przy budowaniu UI
  • To krok 3 z mapy procesu: Design Guidelines

Co robić

  • Krąż, pomagaj z Playwright MCP (konfiguracja)
  • Pokaż na własnym ekranie: jak agent wyciąga tokens
  • Zadbaj, żeby wynik był w repo (katalog design/)

Co sprawdzić

  • Czy Playwright MCP odpala na każdej VM
  • Czy każdy ma min. 3 zrzuty ekranu
  • Czy tokens są zapisane, nie tylko w pamięci agenta
  • Uczestnicy Java: to samo zadanie, design tokens uniwersalne

Moduł 2.2

Komendy i Skills - to samo pod spodem

Custom slash commands zostały scalone ze skills - oba tworzą /x

Jeden plik Markdown

  • .claude/commands/x.md tworzy komendę /x
  • To samo co .claude/skills/x/SKILL.md - oba tworzą /x
  • Świadome, jawne wywołanie /nazwa
  • Wspiera $ARGUMENTS (wstrzykiwanie argumentu)
  • Grupowanie w podfolderach tworzy namespace
Przykład: .claude/commands/git/commit.md wywołuje się jako /git:commit. Drugi przykład: .claude/commands/git/diff.md - /git:diff.

Skill wielo-plikowy

  • Katalog .claude/skills/x/ z SKILL.md + skryptami
  • Agent sam dobiera skill z pola description
  • Auto-trigger kontekstowy (heurystyka opisu)
  • Dołącza skrypty, assets i referencje
  • Metadane określające, kiedy się aktywuje
Kiedy skill wielo-plikowy: gdy agent ma SAM wykryć kontekst, gdy wiedza wymaga wielu plików, gdy potrzebne są dołączone skrypty i assets.
Reguła: prosty, jednokrotny prompt = jeden plik Markdown wywoływany świadomie. Złożony workflow z auto-triggerem i skryptami = skill wielo-plikowy. Pod spodem to ten sam mechanizm - oba tworzą /x.

Moduł 2.2 | Harness

Hooks - deterministyczne bramki jakości

Instrukcja w Markdown

  • Prosi agenta o wykonanie akcji (np. uruchom lint)
  • Zależy od tego, czy agent o niej pomyśli
  • Heurystyka - może zadziałać, może nie
  • Dobra dla wskazówek i preferencji

Hook na zdarzenie

  • Uruchamia akcję deterministycznie po zdarzeniu
  • Np. PostToolUse po edycji wykonanej przez narzędzia Edit|Write - automatycznie lint
  • Niezależne od tego, czy agent o tym pomyśli
  • Gwarancja wykonania, nie prośba
Kontrast: instrukcja prosi o lint; hook uruchamia lint po edycji wykonanej przez narzędzia agenta. Zmiany przez Bash wymagają dodatkowej obsługi. Hook działa niezależnie od tego, czy agent o nim pomyśli.

Moduł 2.3

Architektura sub-agentów

Agent tool (dawniej Task)

  • Mechanizm uruchamiania sub-agenta przez głównego agenta
  • Główny agent deleguje zadanie, sub-agent wykonuje
  • Sub-agent ma własny kontekst i cele
  • Wynik wraca do głównego agenta

Typy agentów

  • Explore - szybki, tylko do odczytu, badanie kodu

Orkiestracja - główny agent kontroluje

Rola głównego agenta

Orchestrator. Rozumie całe zadanie, dzieli na podzadania, deleguje do odpowiednich specjalistów, zbiera wyniki.

Delegowanie

Główny agent wybiera odpowiedniego sub-agenta do zadania. be-dev do backendu, fe-dev do frontendu, qa do testów.

Monitorowanie

Główny agent śledzi postępy sub-agentów, weryfikuje wyniki, reaguje na błędy i problemy.

Feedback loop

Wyniki sub-agenta wracają do głównego agenta. Jeśli coś nie działa, główny agent może poprosić o poprawkę lub przedelegować.

Moduł 2.4 | Krok 6 z mapy procesu

TDD z agentem: Red-Green-Refactor

Cykl TDD

1
Red: napisz test, który nie przechodzi. Agent pisze test PRZED implementacją.
2
Green: napisz minimalny kod, by test przeszedł. Agent implementuje.
3
Refactor: popraw jakość kodu, test nadal przechodzi.

Zasady dla agenta

  • Nie pisz kodu bez testu
  • Nie pisz więcej kodu niż wymaga test
  • Unit: mockuj zależności; integracja: mockuj tylko zewnętrzny LLM; E2E/manual: używaj realnego stosu
  • Test opisuje oczekiwane zachowanie
  • Implementacja jest minimalna
  • Refactor zachowuje zielone testy
  • Realny LLM w E2E i testach manualnych (OpenRouter)

Moduł 2.4 | Wymóg

Git: granularne commity po każdej zielonej fazie

Zasada

  • Osobny commit po spójnej, zielonej i zweryfikowanej zmianie
  • Test i minimalna implementacja trafiają do jednego działającego commitu
  • NIE jeden wielki commit na końcu
  • Historia git = historia implementacji

Czego unikać

  • Jeden commit "wszystko" na końcu dnia
  • Commit z kilkoma funkcjonalnościami naraz
  • Commit bez testów (kod nie zweryfikowany)
  • Brak opisu w komunikacie commitu
# Dobry cykl: # test (Red) → implementacja (Green) → weryfikacja git commit -m "Backend: endpoint analizy zdjęcia"
Dlaczego: granularne commity pozwalają cofnąć się do działającego stanu. Jeśli refactor zepsuje kod, wracasz o jeden commit. Wymóg Łukasza.

Moduł 2.4 | LLM na żywo

OPENROUTER_API_KEY - LLM bez mockowania

Klucz na VM

  • OPENROUTER_API_KEY ustawiony jako zmienna środowiskowa na maszynach kursowych
  • Jeden klucz, wiele modeli, w tym multimodalne
  • Materiały i przykłady kursu używają OpenRouter jako domyślnego providera
  • Nie potrzebujesz własnego klucza API

Bez mockowania

Nic nie jest mockowane. Część LLM aplikacji jest testowana na żywo - zarówno testy E2E, jak i manualne przez Playwright/Chrome DevTools.
