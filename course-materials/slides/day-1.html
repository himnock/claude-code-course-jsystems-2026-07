AI w programowaniu 2026: agenci, nie asystenci

Ewolucja narzędzi

2018
Autocomplete (IntelliSense, TabNine)
2021
Copilot - sugestie w edytorze
2023
Chat z kodem (ChatGPT, Claude)

Zespoły agentów AI
Kierunek
dark factory - trend w stronę autonomicznego procesu wytwarzania oprogramowania, w którym agenci pracują ciągle, a człowiek nadzoruje wyniki

Agent vs asystent

Asystent
  • Sugeruje fragmenty kodu
  • Odpowiada na pytania
  • Człowiek integruje wynik z projektem
Agent
  • Planuje i wykonuje zadania; czyta, pisze, testuje i debuguje kod
  • Weryfikuje swoje działanie, pracuje w pętli
  • Może działać autonomicznie - lokalnie lub na serwerze
  • Wyzwalany zdarzeniami, cronem lub heartbeatem (np. co 30-60 min)
  • Może modyfikować własne ustawienia i pliki konfiguracyjne

Moduł 1.1

Trendy i case studies

Od 1 do wielu

Pojedynczy agent dobrze wykonuje jedno zadanie. Zespół agentów rozdziela role: orchestrator planuje, specjaliści wykonują, krytyk weryfikuje.

Rola człowieka się zmienia

Mniej kodu, więcej architektury i weryfikacji. Trzy nowe role:

  • Orkiestrator - dzieli i opisuje zadania, steruje flotą agentów (nowy tech lead)
  • Budowniczy infrastruktury - CI/CD, MCP, sandboxy, kontrola kosztów
  • Ekspert dziedzinowy - opisuje po ludzku, dostaje działający software

Twój atut: doświadczenie menedżerskie (delegowanie, dzielenie zadań, feedback) = gotowe kompetencje do orkiestracji.

TDD z agentem

Agent pisze testy PRZED implementacją. Unit testy mockują zależności, integracyjne tylko zewnętrzny LLM, a E2E i testy manualne sprawdzają realny stos.

Modele lokalne i open-weight
  • Najlepsze modele open-weight 2026 (GLM-5.2, Kimi K2.6, DeepSeek V4, Minimax M3) są już blisko czołówki - część niemal jak Opus.
  • ALE open != local: najlepsze open-weight są ogromne (setki miliardów parametrów, wymagają serwerowych GPU), nie odpalisz ich na laptopie.
  • Realnie lokalnie (Ollama, vLLM, LM Studio): mniejsze modele jak Gemma 4, Qwen 3.6 - dobre do prywatności / pracy offline (air-gapped), słabsze od modeli chmurowych.
Krajobraz 2026
  • OpenAI - GPT-5.6 Sol/Terra/Luna, ChatGPT Work, Codex
  • Anthropic - Claude Fable 5 (redeploy 01.07)
  • xAI - Grok 4.5, premiera 08.07; początkowo bez dostępu w aplikacjach/API xAI w UE, ale dostępny przez Cursor i OpenRouter; pełny rollout UE oczekiwany w połowie lipca
  • Z.ai - GLM-5.2 + ZCode (1M kontekst, MIT)
  • Google - Antigravity 2.0: desktop app + CLI (zastąpiło Gemini CLI)
  • LMArena - ranking modeli wg głosów ludzi: arena.ai/leaderboard/code/webdev

Moduł 1.1 | Trendy 2026

Agenci asynchroniczni: chmura i mobilność

Delegowanie w tle

Agenci stają się asynchroniczni: delegujesz zadanie w chmurze lub z telefonu i odbierasz gotowy PR później. Nie blokujesz terminala, nie czekasz na odpowiedź.

Agent 24/7 w chmurze

Agent działający ciągle na VPS lub w kontenerze zwiększa autonomię, ale wymaga izolacji, limitów i przeglądu zmian przed produkcją. Bez nadzoru może wprowadzić poważne błędy.

Z telefonu

Delegujesz zadanie z aplikacji mobilnej, agent pracuje w chmurze na Twoim repozytorium. Odbierasz gotowy pull request, gdy jesteś z powrotem przy komputerze.

Implikacja dla zespołu

Asynchroniczność zmienia tryb pracy: z edytora w czasie rzeczywistym na kolejkę zadań i code review gotowych wyników. Nadzór staje się kluczowy.

Multimodalność: screenshoty i HTML jako warstwa prezentacji

Screenshot jako kontekst

Screenshot bywa wygodniejszy do przekazania modelowi układu, kolorów i stanu UI; koszt i jakość zależą od rozdzielczości oraz modelu.

  • Jeden obraz zastępuje akapit opisu UI
  • Model widzi layout, a nie tylko tekst
  • Model nie zawsze poprawnie odczyta drobne szczegóły interfejsu
  • Przydatne przy debugowaniu UI i analizie błędów wizualnych

HTML z Markdown

Markdown zostaje formatem źródłowym, a agent może wygenerować z niego czytelny, interaktywny HTML.

  • PRD, ADR, plan - czytelne w przeglądarce
  • Interaktywne diagramy zamiast statycznego tekstu
  • Źródło w git (Markdown), prezentacja w HTML
  • Dokumentacja żyje razem z kodem
Na tym kursie: PRD i ADR piszemy w Markdown, a ta prezentacja jest generowanym HTML-em. Zobaczysz ten wzorzec wielokrotnie.

Moduł 1.1

Korzyści vs ryzyka

Korzyści

  • Szybkość dostarczania - scaffolding, boilerplate, refaktoryzacja w minuty
  • Mniej powtarzalnej pracy - agent przejmuje CRUD, testy, dokumentację
  • Nauka nowych wzorców - agent pokazuje rozwiązania, które sam byś przegapił
  • Eksploracja - szybkie prototypowanie pomysłów
  • Dokumentacja na bieżąco - CLAUDE.md, ADR, komentarze generowane w locie

Ryzyka

  • Więcej bugów i rework - kod powstaje szybciej, ale badania notują więcej defektów i poprawek na produkcji, co zjada zyski z prędkości
  • Więcej kodu ≠ więcej wartości - generujemy więcej, rośnie dług techniczny, legacy i koszt utrzymania
  • Halucynacje - agent zmyśla API, funkcje, biblioteki, które nie istnieją
  • Context rot - w długich sesjach agent traci kontekst, gubi wymagania
  • Koszty - tokeny, wywołania API, mogą rosnąć niekontrolowanie
  • Fałszywe poczucie bezpieczeństwa - kod wygląda dobrze, ale może mieć subtelne błędy
  • Zależność - bez agenta trudniej pracować (atrofia umiejętności)
Mitygacja: weryfikacja, testy, code review, świadome zarządzanie kontekstem. O tym cały kurs.
Co mówią badania:
  • Krzywa J - produktywność spada na 1-2 miesiące przy nauce nowego workflow (dane głównie z 2025; dziś modele dużo lepsze)
  • Zanik pamięci mięśniowej do ręcznego kodu jest realny - ale czy przy generowaniu musimy jeszcze pisać wszystko ręcznie?
  • Mierz własne ROI: velocity, liczba bugów, poprawki i komentarze w code review
  • Źródła: Stanford (100k+ dev) · METR 2025 · SSRN: ukryte koszty GenAI · arXiv 2025 (Indie)

Zarządzanie kontekstem

Okno kontekstu

Ile agent może pamiętać na raz w jednej sesji. Im większe okno, tym więcej plików i historii się mieści.

Claude Fable / Opus / Sonnet: 1M tokenów; Haiku: 200K; GLM-5.2: 1M. Uwaga: polski tekst zużywa więcej tokenów niż angielski.

Streszcza historię sesji. Zachowuje kluczowe decyzje i pliki, odrzuca detale rozmowy. Przedłuża żywotność sesji.

Zjawisko degradacji jakości w długich sesjach. Agent zaczyna gubić wątki, zapomina wcześniejsze decyzje, odpowiada coraz ogólnikowo.

Strategie
  • Dziel duże zadania na krótsze sesje
  • Używaj CLAUDE.md jako trwałej pamięci
  • Stosuj /compact regularnie
  • Zaczynaj nową sesję dla nowego zadania

Uprawnienia: tryby pracy agenta

Shift+Tab domyślnie przełącza: default → acceptEdits → plan; aktualny tryb widać w pasku stanu

Tryby uprawnień

  • default - odczyt bez pytania; edycje, komendy i sieć według reguł uprawnień
  • accept edits - automatycznie akceptuje edycje plików, o resztę pyta
  • plan - agent tylko czyta i planuje; zmiany dopiero po Twoim zatwierdzeniu
  • auto - dostępny po spełnieniu dodatkowych warunków; klasyfikator ocenia ryzyko akcji
  • bypass permissions - pełne pominięcie pytań o zgodę; tylko w izolowanym środowisku i po świadomym włączeniu

Reguły w settings.json

allow / ask / deny per narzędzie lub komenda - np. deny na rm i deploy, allow na testy i lint. Edytuj przez /permissions.

Moduł 1.3

Od prompt engineering do context engineering

Prompt engineering

Jak sformułować pojedyncze zapytanie, żeby dostać dobrą odpowiedź.

  • "Zrefaktoruj ten moduł na async/await; publiczne API bez zmian, testy mają przechodzić"
  • "POST /orders zwraca 500 przy pustym body - znajdź przyczynę, napraw i dodaj test regresyjny"
  • Cały potrzebny kontekst upychasz w jednym prompcie - za każdym razem od nowa

Context engineering

Jak zbudować całe środowisko agenta, żeby systematycznie dostawać dobre wyniki.

  • CLAUDE.md i AGENTS.md jako instrukcje
  • Konwencje, zasady, stack
  • Struktura projektu opisana dla agenta
  • Trwała pamięć między sesjami
Kluczowa różnica: prompt engineering = pojedyncze zapytanie. Context engineering = środowisko, w którym agent działa i pracuje.

Path-based rules vs nested CLAUDE.md

Nested CLAUDE.md

Pliki CLAUDE.md w podkatalogach projektu.

Gdy agent wchodzi do katalogu, czyta tamtejszy CLAUDE.md.

Reguły to pliki Markdown w .claude/rules/ z polem paths: we frontmatter.

# .claude/rules/frontend.md --- paths: - "frontend/**/*" --- Używaj komponentów funkcyjnych React.

Best practices: jak pisać CLAUDE.md

Krótko i konkretnie

Celuj w około 200 linii jako zalecenie kursowe, nie limit techniczny. Pełny CLAUDE.md jest ładowany niezależnie od długości.

Konkretne reguły

Źle: "Pisz czysty kod". Dobrze: "Używaj TypeScript strict mode, wszystkie funkcje muszą mieć typy zwracane."

Iteruj

Dodaj regułę za każdym razem, gdy agent popełni błąd. CLAUDE.md to żywy dokument.

Wersjonuj w git

CLAUDE.md w repozytorium = cała drużyna ma te same reguły. Historia zmian widoczna.

# CLAUDE.md - przykład ## Stack TypeScript, Next.js, Tailwind CSS, PostgreSQL ## Konwencje - Używaj TypeScript strict mode - Komponenty funkcyjne, nie class components - Nazwy plików: kebab-case ## Testy - Testy PRZED implementacją (TDD) - Unit: mocki zależności; integracja: mock tylko zewnętrznego LLM; E2E: realny stos

Architektura: TypeScript + Vercel AI SDK + streaming

TypeScript

Type safety na froncie i backendie. Mniej błędów runtime, lepsze DX, agent ma jasny kontrakt typów.

Vercel AI SDK

Abstrakcja nad LLM API. Ujednolica dostęp do różnych modeli. Wbudowane wsparcie dla streaming, tool calling, multimodalności.

Streaming

Odpowiedzi w czasie rzeczywistym. Użytkownik widzi tekst pojawiający się słowo po słowie, zamiast czekać na pełną odpowiedź.

Template Vercel AI

Agent inicjalizuje projekt z gotowego template. Scaffolding w minuty, nie godzinach. My skupiamy się na logice biznesowej.

Moduł 1.4

Dostęp aplikacji do LLM: OPENROUTER_API_KEY

Co to jest?

  • Zmienna środowiskowa OPENROUTER_API_KEY już ustawiona na VM
  • Jeden klucz, wiele modeli - nie potrzebujesz osobnych kont
  • Modele multimodalne dostępne (analiza zdjęć)
  • Preset kursowy dla aplikacji - gotowy do użycia od razu

Jak sprawdzić

echo %OPENROUTER_API_KEY%

Lub w PowerShell:

$env:OPENROUTER_API_KEY
To klucz aplikacji, nie Claude Code. Claude Code uwierzytelniasz własnym kontem/subskrypcją Claude albo przez ANTHROPIC_API_KEY.
