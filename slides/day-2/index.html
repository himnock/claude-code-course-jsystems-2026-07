Moduł 2.1

Czym jest MCP

Model Context Protocol - protokół komunikacji agenta z zewnętrznymi narzędziami

Zasada działania

  • MCP server wystawia narzędzia (tools) dla agenta
  • Agent wywołuje narzędzia w trakcie pracy, gdy ich potrzebuje
  • Serwer działa jako osobny proces, stale dostępny
  • Konfiguracja w pliku .mcp.json lub settings

Do czego służy

  • Dostęp do dokumentacji bibliotek (Context7)
  • Sterowanie przeglądarką (Playwright MCP)
  • Integracja z bazami danych, API zewnętrznymi
  • Każde narzędzie, które agent może wywołać programowo
Idea: zamiast wklejać dokumentację w prompt, agent sam pobiera aktualną dokumentację przez MCP, gdy jej potrzebuje.

Moduł 2.1

Dlaczego coraz częściej Skills zamiast MCP

MCP - kiedy jeszcze sensowne

  • Stała integracja z usługą (baza danych, API)
  • Narzędzie, którego używasz codziennie, przez cały projekt
  • Playwright MCP - sterowanie przeglądarką w E2E
  • Context7 - dokumentacja na żądanie

Skills - kiedy lepsze

  • Powtarzalny workflow (ciąg kroków, nie jedno narzędzie)
  • Bundle instrukcji + skryptów w jednym miejscu
  • Łatwiejsze do wersjonowania i udostępniania
  • Mniej narzut niż utrzymywanie serwera MCP
Trend: agenty przechodzą od "stałych serwerów MCP" do "Skills + CLI wywoływane na żądanie". MCP zostaje dla trwałych integracji, Skills przejmują workflows.

Moduł 2.1

Konfiguracja MCP: .mcp.json i settings

Plik .mcp.json

Konfiguracja serwerów MCP w projekcie. Definiuje, które serwery są dostępne i jak je uruchomić.

{ "mcpServers": { "context7": { "command": "npx", "args": ["-y", "@context7/mcp"] }, "playwright": { "command": "npx", "args": ["-y", "@playwright/mcp"] } } }

Poziomy konfiguracji

  • .mcp.json w katalogu projektu - per projekt
  • Settings globalne - dostępne w każdym projekcie
  • Agent wczytuje konfigurację przy starcie
Sprawdź: użyj komendy listującej aktywne serwery MCP w swoim CLI, aby zweryfikować że konfiguracja zadziałała.

Moduł 2.1 | Narzędzia

Playwright MCP - przeglądarka w rękach agenta

Sterowanie przeglądarką

Agent otwiera strony, klika, czyta DOM, robi zrzuty ekranu. Pełna automatyzacja przeglądarki przez MCP.

Design Guidelines

Reverse-engineering designu dowolnej strony: kolory, typografia, układy. Wyciąganie design tokens do własnej aplikacji.

Testy E2E

Automatyczne testy end-to-end. Agent wchodzi na aplikację, klika, weryfikuje czy funkcja działa. Bez mockowania.

Manualne testy agenta

Agent testuje aplikację jak człowiek: otwiera, wchodzi przez flow, sprawdza rezultaty. Nic nie mockujemy - LLM testowany na żywo.

W naszym procesie: Playwright MCP jest kluczowe na dwóch etapach - Design Guidelines (lab 2.1) oraz testy manualne aplikacji (moduł 2.4 i dzień 3).

Moduł 2.1 | Ważny trend

CLI rośnie vs MCP

Klasycznie: serwer MCP stale włączony

  • Serwer MCP działa cały czas, nawet gdy niepotrzebny
  • Narzut: proces, pamięć, konfiguracja utrzymania
  • Dobre dla integracji, które używasz non-stop

Trend: CLI wywoływane przez agenta

  • Agent wywołuje narzędzie CLI na żądanie, gdy potrzebuje
  • Brak stale działającego serwera - mniej narzutu
  • Narzędzie startuje, robi robotę, kończy
  • Łatwiejsze debugowanie i izolacja
Praktyczna konsekwencja: coraz więcej narzędzi dostępnych jako CLI wywoływane przez agenta, zamiast trwale skonfigurowanego serwera MCP. Wybieraj świadomie - MCP dla stałych integracji, CLI/Skills dla jednorazowych i workflow.

Inne serwery i integracje MCP

Bazy danych

Bezpośredni dostęp do PostgreSQL, SQLite, innych. Agent czyta schemat, pisze zapytania, weryfikuje dane.

API zewnętrzne

Integracja z REST/GraphQL APIs. Agent wywołuje endpointy, czyta odpowiedzi, testuje kontrakty.

Git i GitHub

Operacje na repozytorium, PR, issues. Agent może tworzyć commity, otwierać PR, czytać review.

System plików

Czytanie, zapis, wyszukiwanie plików. Podstawa pracy agenta z kodem.

Na tym kursie skupiamy się na Context7 i Playwright MCP. Pozostałe jako przegląd - wiadomość, że MCP jest ekosystemem, nie pojedynczym narzędziem.

Skrypty, narzędzia i custom slash commands

Skrypty w skills

Skrypt bash/python/node dołączony do skilla. Agent wywołuje go w odpowiednim momencie workflow. Automatyzuje powtarzalne operacje.

Narzędzia w skills

Skill może bundled'ować narzędzia CLI. Agent ma wszystko w jednym miejscu - instrukcje + narzędzia + szablony.

Custom slash commands

Własne komendy / definiowane w skillu. Np. /design-review uruchamia pełny workflow design review. Skrót do całego procesu.

Architektura sub-agentów

  • Główny agent deleguje zadanie, sub-agent wykonuje
  • Sub-agent ma własny kontekst i cele

Typy agentów

  • Custom - specjaliści: be-dev, fe-dev, qa, itd.

Moduł 2.3

Orkiestracja - główny agent kontroluje

Rola głównego agenta

Orchestrator. Rozumie całe zadanie, dzieli na podzadania, deleguje do odpowiednich specjalistów, zbiera wyniki.

Delegowanie

Główny agent wybiera odpowiedniego sub-agenta do zadania. be-dev do backendu, fe-dev do frontendu, qa do testów.

Monitorowanie

Główny agent śledzi postępy sub-agentów, weryfikuje wyniki, reaguje na błędy i problemy.

Feedback loop

Wyniki sub-agenta wracają do głównego agenta. Jeśli coś nie działa, główny agent może poprosić o poprawkę lub przedelegować.

Metafora: główny agent to tech lead. Sub-agenci to członkowie zespołu. Tech lead nie pisze całego kodu - deleguje i weryfikuje.

Moduł 2.4 | Krok 6 z mapy procesu

TDD z agentem: Red-Green-Refactor

Cykl TDD

1
Red: napisz test, który nie przechodzi. Agent pisze test PRZED implementacją.
2
Green: napisz minimalny kod, by test przeszedł. Agent implementuje.
3
Refactor: popraw jakość kodu, test nadal przechodzi.

Zasady dla agenta

  • Nie pisz kodu bez testu
  • Nie pisz więcej kodu niż wymaga test
  • Nie mockuj - testuj na żywo
  • Test opisuje oczekiwane zachowanie
  • Implementacja jest minimalna
  • Refactor zachowuje zielone testy
  • LLM testowany na żywo (OpenRouter)

Moduł 2.4 | Wymóg

Git: granularne commity po każdej zielonej fazie

Zasada

  • Osobny commit po każdym przechodzącym teście
  • Commit po każdej zielonej fazie TDD
  • NIE jeden wielki commit na końcu
  • Historia git = historia implementacji

  • Jeden commit "wszystko" na końcu dnia
  • Commit z kilkoma funkcjonalnościami naraz
  • Commit bez testów (kod nie zweryfikowany)
  • Brak opisu w komunikacie commitu

Dlaczego: granularne commity pozwalają cofnąć się do działającego stanu. Jeśli refactor zepsuje kod, wracasz o jeden commit. Wymóg Lucasa.
