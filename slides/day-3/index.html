3.1 - narzędzie

Git worktrees - równoległe środowiska

Git worktree pozwala mieć wiele checkout-ów tego samego repozytorium w osobnych katalogach. Każdy agent pracuje w swoim katalogu bez blokowania innych.

# Tworzenie worktree dla agenta backend git worktree add ../project-be feature/backend-auth # Tworzenie worktree dla agenta frontend git worktree add ../project-fe feature/frontend-chat # Tworzenie worktree dla agenta testów E2E git worktree add ../project-e2e feature/e2e-tests # Lista aktywnych worktrees git worktree list
Zalety worktrees
  • Każdy agent ma izolowane środowisko plików
  • Nie trzeba klonować repo wielokrotnie
  • Współdzielona historia git (jeden .git)
  • Łatwy merge z powrotem do main
Na co uważać
  • Ten sam branch nie może być checkout w dwóch worktrees
  • Należy czyścić worktrees po merge (git worktree remove)
  • Zależności (node_modules) muszą być instalowane osobno

3.1 - taksonomia automatyzacji

Workflow - cztery mechanizmy

Workflow to szerokie pojęcie: może być batchem sub-agentów, hookiem, pętlą celu albo zadaniem cyklicznym - wybierz mechanizm do charakteru pracy.

Batch scripts

Skrypt orkiestruje wielu sub-agentów naraz, np. CSV z listą zadań - każdy wiersz to osobne uruchomienie z własnym kontekstem.

Hooks

Deterministyczne zdarzenia: agent reaguje na moment, np. po zapisie pliku lub przed committem - powtarzalnie i bez ręcznego startu.

Goals / loops

Agent dostaje cel wysokiego poziomu, sam planuje kroki i działa w pętli aż cel zostanie osiągnięty - najbardziej autonomiczny wariant.

Schedules / cron

Zadanie cykliczne uruchamiane bez ręcznego startu - np. codzienny review zmian albo nocne odpalenie batcha.

Skalowanie: to kontrolowany batch i limit współbieżności, nie bezwarunkowe uruchomienie setek agentów naraz. Limituj też głębokość zagnieżdżenia sub-agentów.

Co mówić

  • Workflow to kategoria, nie jedna komenda - pokaż cztery warianty obok siebie
  • Batch = wielu agentów jednorazowo, hook = reakcja na zdarzenie, goal = pętla autonomiczna, schedule = cyklicznie
  • Skalowanie NIE polega na odpaleniu setek agentów naraz - batch + limit współbieżności + limit głębokości

Co robić

  • Pokaż przykład CSV z listą zadań jako wejście do batcha
  • Nawiąż do hooków z dnia 2 (deterministyczne zdarzenia, permissions = prawo)

Co sprawdzić

  • Czy uczestnicy widzą różnicę między goal/loop a zwykłym batchem

Akt 3 (15 min) - najcenniejsza część

Wpadki - część 1

Prawdziwe awarie z tej misji i czego uczą. Każda to lekcja, nie tylko anegdota.

1. Zombie proces

Model uruchomiony w tle zawiesił się z zerowym outputem. Plik raportu miał 0 bajtów.

Lekcja: weryfikuj OUTPUT, nie exit code. Preferuj foreground z jawnym timeoutem.

2. Sandbox vs git

Model-wykonawca nie mógł zrobić commita (blokada w sandboxie, problem właściciela repo w innym środowisku).

Lekcja: commituje orkiestrator albo skrypt, nigdy model w sandboxie. Sandbox to feature bezpieczeństwa, nie bug do obejścia.

3. Model pinning w headless

CLI listuje wiele modeli interaktywnie, ale w trybie headless flaga wyboru modelu bywa IGNOROWANA - dostajesz inny (tańszy/szybszy) model niż oczekiwany.

Lekcja: zawsze weryfikuj który model FAKTYCZNIE odpowiada (zapytaj go wprost) przed zdelegowaniem poważnej pracy.

Akt 3 (ciąg dalszy)

Wpadki - część 2

4. Odporność na prompt injection

Orkiestrator dosłał subagentowi dodatkową instrukcję W TRAKCIE jego pracy (przez output narzędzia, nie bezpośrednio od użytkownika). Subagent uznał to za możliwą próbę prompt injection i ODMÓWIŁ wykonania, flagując to w raporcie.

Lekcja: słuszna paranoja, mniej wygodny wynik - i tak WŁAŚNIE ma wyglądać dobra defensywa modelu.

5. Kwoty i budżety

W nocy skończyła się kwota jednego z modeli-wykonawców do określonej godziny.

Rozwiązanie: skrypt w tle czekał do reaktywacji kwoty, zaktualizował CLI, zrobił smoke test nowego modelu i dokończył pracę - nikt nie czuwał ręcznie.

6. Guardrail godzinowy (opcjonalnie)

Agent ma twardą zasadę odmawiania pracy interaktywnej po określonej porze w nocy - i użył jej wobec własnego użytkownika.

Lekcja: automatyzacja obejmuje też higienę pracy człowieka, nie tylko kod.

Co mówić

  • To jest NAJCENNIEJSZA część demo - prawdziwe wpadki są warte więcej niż sukcesy
  • Każda wpadka to lekcja - podkreśl co się nauczyliśmy

Co robić

  • Pokaż plik raportu 0 bajtów (zombie) jeśli jeszcze istnieje
  • Pokaż raport z odmową prompt injection

Co sprawdzić

  • Czy uczestnicy rozumieją że weryfikacja output > weryfikacja exit code

3.2 - zasada jakościowa (TWARDA REGUŁA)

Testy manualne agenta: NIC nie jest mockowane

To nie jest sugestia. To jest twarda reguła tego kursu i tej firmy.

Kiedy agent testuje aplikację przez Playwright MCP lub Chrome DevTools, testy są PRAWDZIWE:

  • NIE są symulacją ani nagraniem
  • NIE używamy mocków dla wywołań modelu
  • NIE testujemy na zastępczym środowisku
  • TAK - prawdziwe wywołania modelu (live LLM calls)
  • TAK - prawdziwa przeglądarka (Chrome/Chromium przez MCP)
  • TAK - prawdziwe API, prawdziwa baza danych, prawdziwe odpowiedzi
Dlaczego to ma znaczenie: mockowany test udaje że wszystko działa. Prawdziwy test wyłapuje co naprawdę się psuje. Jeśli testujesz agenta na mockach, nie wiesz czy agent faktycznie radzi sobie z prawdziwym zachowaniem modelu. To byłoby oszustwo wobec siebie i wobec uczestników.

Co mówić

  • PODKREŚL TO MOCNO - to jest zasada jakościowa tej firmy
  • "Twarda reguła, nie sugestia" - dosłownie to powiedz
  • Wyjaśnij dlaczego mocki są niebezpieczne: udają że działa

Co robić

  • FAKTYCZNIE zademonstruj test na żywo: agent przez Playwright MCP klika w prawdziwą aplikację, wysyła prawdziwe zapytanie do API, dostaje prawdziwą odpowiedź od modelu
  • Pokaż w terminalu że wywołania modelu są realne (logi, koszty tokenów)

Co sprawdzić

  • Czy uczestnicy widzą że to jest NA ŻYWO (nie nagranie)
  • Czy rozumieją różnicę między mockiem a prawdziwym testem

3.3 - CI/CD

Headless -p w GitHub Actions

Flaga -p (print) uruchamia Claude Code w trybie headless - jedno zapytanie, jedna odpowiedź, koniec. Idealne do automatyzacji w CI/CD.

name: AI Code Review on: [pull_request] jobs: review: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - name: Install Claude Code run: npm install -g @anthropic-ai/claude-code - name: AI Review env: ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }} run: | claude -p "Przejrzyj zmiany w tym \ pull request i wskaż ryzyka \ bezpieczeństwa oraz problemy \ jakości kodu" \ --output-format json > review.json - name: Post review comment run: | cat review.json | jq '.response' \ > comment.md gh pr comment ${{ github.event.pull_request.number }} \ --body-file comment.md

3.3 - CI/CD wariant

Wariant Jenkins

Fakt z ankiety uczestników: 3 na 4 przebadanych uczestników używa Jenkinsa (nie GitHub Actions) w swojej pracy. Dlatego pokazujemy jak analogicznie wywołać Claude Code headless z poziomu Jenkinsa.
// Jenkinsfile - AI Code Review pipeline { agent any environment { CLAUDE_API_KEY = credentials('claude-api-key') // [TODO-VERIFY: dokładna składnia withCredentials vs environment credentials w Jenkins Pipeline] } stages { stage('AI Review') { steps { sh '''claude -p "Przejrzyj zmiany w \ tym pull request i wskaż ryzyka" \ --output-format json > review.json''' // [TODO-VERIFY: czy wymagany dodatkowy plugin do uruchamiania CLI w kontenerze Jenkins agenta] } } } }
Różnice vs GitHub Actions
  • Runner: self-hosted agent/executor zamiast hostowanego runnera GHA
  • Sekrety: Jenkins Credentials Binding zamiast GitHub Secrets
  • Wykonanie: sh step w Pipeline zamiast run w YAML
To samo wywołanie

Sama komenda claude -p "..." jest identyczna niezależnie od platformy CI/CD. Różni się tylko otoczenie: jak definiujesz sekrety, jak uruchamiasz kroki, gdzie logujesz wyniki.

Co mówić

  • Podkreśl fakt z ankiety: 3/4 uczestników używa Jenkinsa
  • To nie jest "Jenkins vs GitHub Actions" - to pokazanie że Claude działa w obu
  • Sekrety przez Jenkins Credentials, nie hardcoded
  • TODO-VERIFY oznacza fragmenty gdzie dokładna składnia Jenkins wymaga weryfikacji - bądź uczciwy

Co robić

  • Pokaż Jenkinsfile obok pliku workflow YAML z GHA - porównanie
  • Zaznacz że sama komenda claude -p jest taka sama

Co sprawdzić

  • Czy uczestnicy rozumieją mapowanie: GHA secrets = Jenkins credentials
  • Czy ktoś używa innego CI/CD (GitLab CI, CircleCI)

3.3 - ekonomia

Koszty i optymalizacja

Tokeny

Największy koszt to tokeny modelu-orkiestratora. Każde wywołanie headless zużywa tokeny. Monitoruj zużycie w panelu billingowym Anthropic.

Modele

Orkiestrator: drogi, inteligentny model. Wykonawcy: tańsze modele. Wybór modelu to tradeoff między jakością a kosztem.

Cache

Prompt caching redukuje koszty powtarzalnych wywołań. Instrukcje systemowe, kontekst projektu, CLAUDE.md - wszystko co się powtarza powinno być cache-owane.

Strategie optymalizacji

  • Używaj tańszych modeli do prostych zadań (review, formatowanie)
  • Drogi model tylko do orkiestracji i weryfikacji
  • Cache-uj kontekst który się powtarza
  • Ograniczaj kontekst: raport zamiast surowych plików
  • Monitoruj zużycie - ustaw alerty budżetowe

Czego unikać

  • Wrzucanie całego repo jako kontekst do każdego wywołania
  • Używanie najdroższego modelu do wszystkiego
  • Brak limitów budżetowych w CI/CD (partyjka kosztów)
  • Ignorowanie cache - płacisz wielokrotnie za to samo

n8n - automatyzacja bez kodu

n8n to platforma automatyzacji workflow no-code/low-code. Możesz używać jej jako alternatywnego lub uzupełniającego narzędzia do orkiestracji agentów - wizualny edytor przepływów, integracje z setkami usług, webhook-i i harmonogramy.

Modele lokalne

Trzy główne runtimes - wybór zależy od sprzętu i obciążenia:

  • Ollama - najprostszy start lokalny, dobra na maszynę deweloperską i szybkie eksperymenty
  • vLLM - skaluje współbieżność, gdy jeden model obsługuje wielu użytkowników lub agentów naraz
  • LocalAI - wystawia API zgodne z OpenAI, drop-in zamiennik dla kodu pisanego pod chmurę
Lokalny model umożliwia pracę w środowisku air-gapped, ale modele, zależności i dokumentację trzeba dostarczyć wcześniej - to nie jest zero-setup.

To są kierunki rozwoju po kursie. Wybierz jeden i zgłębiaj go.

Co mówić

  • Modele lokalne = prywatność i brak kosztów krańcowych, ale hardware ma znaczenie
  • Ollama = start, vLLM = obciążenie/współbieżność, LocalAI = zgodność API z OpenAI
  • Air-gapped: modele, zależności i dokumentację dostarcz wcześniej - to nie zero-setup

  • Weryfikuj dostępność narzędzi przed każdym terminem kursu - to nie są fakty do zapamiętania, tylko sygnał że teren się rusza
