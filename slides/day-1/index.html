Moduł 1.1

Trendy i case studies

Od 1 do wielu

Pojedynczy agent dobrze wykonuje jedno zadanie. Zespół agentów rozdziela role: orchestrator planuje, specjaliści wykonują, krytyk weryfikuje.

Rola człowieka się zmienia

Programista coraz częściej pełni rolę architekta i recenzenta, a nie tylko piszącego kod. Kod generuje agent, człowiek weryfikuje i kieruje.

TDD z agentem

Agent pisze testy PRZED implementacją. Testuje manualnie przez Playwright/Chrome DevTools. Nic nie jest mockowane - LLM testowany na żywo.

Krajobraz 2026
  • OpenAI - GPT-5.6 Sol/Terra/Luna, ChatGPT Work, Codex
  • Anthropic - Claude Fable 5 (redeploy 01.07)

  • Z.ai - GLM-5.2 + ZCode (1M kontekst, MIT)

Moduł 1.1 | Trendy 2026

Agenci asynchroniczni: chmura i mobilność

Delegowanie w tle

Agenci stają się asynchroniczni: delegujesz zadanie w chmurze lub z telefonu i odbierasz gotowy PR później. Nie blokujesz terminala, nie czekasz na odpowiedź.

Agent 24/7 w chmurze

Agent działający ciągle na VPS lub w kontenerze zwiększa autonomię, ale wymaga izolacji, limitów i przeglądu zmian przed produkcją. Bez nadzoru może narobić szkód.

Z telefonu

Delegujesz zadanie z aplikacji mobilnej, agent pracuje w chmurze na Twoim repozytorium. Odbierasz gotowy pull request, gdy jesteś z powrotem przy komputerze.

Implikacja dla zespołu

Asynchroniczność zmienia tryb pracy: z edytora w czasie rzeczywistym na kolejkę zadań i code review gotowych wyników. Nadzór staje się kluczowy.

Moduł 1.1 | Trendy 2026

Multimodalność: screenshoty i HTML jako warstwa prezentacji

Screenshot jako kontekst

Screenshot potrafi przekazać modelowi układ, kolory i treść taniej niż długi opis tekstowy.

  • Jeden obraz zastępuje akapit opisu UI
  • Model widzi layout, a nie tylko tekst
  • Kompresja kontekstu: obraz > tysiące słów
  • Przydatne przy debugowaniu UI i analizie błędów wizualnych

HTML z Markdown

Markdown zostaje formatem źródłowym, a agent może wygenerować z niego czytelny, interaktywny HTML.

  • PRD, ADR, plan - czytelne w przeglądarce
  • Interaktywne diagramy zamiast statycznego tekstu
  • Źródło w git (Markdown), prezentacja w HTML
  • Dokumentacja żyje razem z kodem
Na tym kursie: PRD i ADR piszemy w Markdown, a ta prezentacja jest generowanym HTML-em. Zobaczysz ten wzorzec wielokrotnie.

Moduł 1.1

Korzyści vs ryzyka

Korzyści

  • Szybkość dostarczania - scaffolding, boilerplate, refaktoryzacja w minuty
  • Mniej powtarzalnej pracy - agent przejmuje CRUD, testy, dokumentację
  • Nauka nowych wzorców - agent pokazuje rozwiązania, które sam byś przegapił
  • Eksploracja - szybkie prototypowanie pomysłów
  • Dokumentacja na bieżąco - CLAUDE.md, ADR, komentarze generowane w locie

Ryzyka

  • Halucynacje - agent zmyśla API, funkcje, biblioteki, które nie istnieją
  • Context rot - w długich sesjach agent traci kontekst, gubi wymagania
  • Koszty - tokeny, wywołania API, mogą rosnąć niekontrolowanie
  • Fałszywe poczucie bezpieczeństwa - kod wygląda dobrze, ale może mieć subtelne błędy
  • Zależność - bez agenta trudniej pracować (atrofia umiejętności)
Mitygacja: weryfikacja, testy, code review, świadome zarządzanie kontekstem. O tym cały kurs.

CLI interaktywny vs tryb -p

CLI interaktywny

  • Sesja dwukierunkowa, agent pamięta kontekst
  • Komendy / dostępne (/model, /compact itd.)
  • Można wglądać w pliki, edytować, iterować
  • Idealny do codziennej pracy i developmentu
claude

Tryb -p (print)

  • Jednorazowe zapytanie, wynik do stdout
  • Brak sesji, brak kontekstu z poprzednich zapytań
  • Można pipe'ować (wejście/wyjście)
  • Idealny do skryptów, CI/CD, automatyzacji

Moduł 1.2 | Kluczowe komendy (1/2)

Zarządzanie sesją i kontekstem

/resume

Wznawia poprzednią sesję. Agent odtwarza kontekst, historię rozmowy i stan projektu.

Przydatne gdy przerwałeś pracę i chcesz wrócić, lub gdy przypadkowo zamknąłeś CLI.

/compact

Kompresuje kontekst sesji. Agent streszcza dotychczasową rozmowę, zachowując kluczowe informacje, a pomijając detale.

Stosuj gdy sesja robi się długa i agent zaczyna gubić wątki (context rot).

Wskazówka: /compact nie usuwa plików ani zmian w kodzie. Kompresuje tylko historię rozmowy w kontekście agenta.

Diagnostyka, modele i uprawnienia

/context

Pokazuje strukturę kontekstu agenta: co widzi (system prompt, CLAUDE.md, otwarte pliki, historia sesji).

Używaj gdy agent dziwnie odpowiada - sprawdź co naprawdę wie.

/model

Przełącza model (Opus / Sonnet / Haiku). Wyższy model = lepsze wyniki, ale drożej i wolniej.

Używaj gdy potrzebujesz innej jakości/kosztu do konkretnego zadania.

/permissions

Zarządzanie uprawnieniami: allowedTools (co może bez pytania), denyTools (co nie może nigdy).

Używaj gdy chcesz kontrolować co agent może robić z Twoim systemem.

Zarządzanie kontekstem

Okno kontekstu

Ile agent może pamiętać na raz w jednej sesji. Im większe okno, tym więcej plików i historii mieści się.

GLM-5.2: 1M tokenów (potwierdzone). [TODO-VERIFY: okno kontekstu dla pozostałych modeli]

Kompresja (/compact)

Streszcza historię sesji. Zachowuje kluczowe decyzje i pliki, odrzuca detale rozmowy. Przedłuża żywotność sesji.

Context rot

Zjawisko degradacji jakości w długich sesjach. Agent zaczyna gubić wątki, zapomina wcześniejsze decyzje, odpowiada coraz ogólnikowo.

Strategie
  • Dziel duże zadania na krótsze sesje
  • Używaj CLAUDE.md jako trwałej pamięci
  • Stosuj /compact regularnie
  • Zaczynaj nową sesję dla nowego zadania

Moduł 1.3

Od prompt engineering do context engineering

Prompt engineering

Jak sformułować pojedyncze zapytanie, żeby dostać dobrą odpowiedź.

  • "Napisz funkcję, która..."
  • "Wyjaśnij ten błąd"
  • Jednorazowa interakcja

Context engineering

Jak zbudować całe środowisko agenta, żeby systematycznie dostawać dobre wyniki.

  • CLAUDE.md i AGENTS.md jako instrukcje
  • Konwencje, zasady, stack
  • Struktura projektu opisana dla agenta
  • Trwała pamięca między sesjami
Kluczowa różnica: prompt engineering = pojedyncze zapytanie. Context engineering = środowisko, w którym agent żyje i pracuje.

Moduł 1.3

Path-based rules vs nested CLAUDE.md

Nested CLAUDE.md

Pliki CLAUDE.md w podkatalogach projektu.

Gdy agent wchodzi do katalogu, czyta tamtejszy CLAUDE.md.

projekt/ CLAUDE.md # globalne reguły frontend/ CLAUDE.md # reguły dla FE backend/ CLAUDE.md # reguły dla BE

Path-based rules

Reguły zależne od katalogu, definiowane w głównym pliku konfiguracyjnym.

Zamiast wielu plików, reguły w jednym miejscu z warunkami na ścieżkę.

Hierarchia czytania: global -> root -> subdir. Każdy poziom może nadpisać lub rozszerzyć poprzedni.

Wybór: nested dla dużych projektów z wyraźnymi modułami. Path-based dla mniejszych projektów z specyficznymi regułami per katalog.
